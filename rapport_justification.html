1. Contexte & Architecture Technique

L'objectif du TP4 est d'industrialiser le déploiement de l'application Spring PetClinic sur AWS en appliquant les principes GitOps et CI/CD. L'infrastructure est entièrement déclarée sous forme de code (Infrastructure as Code) avec Terraform et hébergée sur la région us-east-1.

Composants de l'Infrastructure AWS (Multi-AZ)

  • Réseau (VPC Multi-AZ) : Découpé en sous-réseaux Publics (ALB, NAT Gateways), Privés Application (Auto Scaling Group) et Privés Base de Données (RDS MySQL).
  • Application Load Balancer (ALB) : Point d'entrée HTTPS sécurisé avec redirection HTTP et terminaison SSL/TLS.
  • Auto Scaling Group (ASG) : Gestion dynamique de 2 à 4 instances EC2 avec politique de mise à l'échelle automatique basée sur l'utilisation CPU.
  • Base de Données RDS MySQL Multi-AZ : Haute disponibilité avec réplication synchrone et basculement automatique.
  • Sécurité & Secrets : AWS Secrets Manager et KMS pour le stockage et chiffrement des identifiants sans aucun secret en dur.

2. Pipeline CI/CD & Principes GitOps

Le pipeline est orchestré par GitHub Actions et s'exécute automatiquement à chaque événement sur la branche principale :

  1. Build & Test : Compilation de l'application Spring Boot via ./mvnw clean package -DskipTests et génération du JAR.
  2. Security Gates : Contrôle de qualité du code via Checkstyle / NoHTTP.
  3. Terraform Init / Plan / Apply : Déploiement automatisé des ressources AWS déclarées.
  4. Smoke Test & Healthcheck : Validation opérationnelle sur l'endpoint /actuator/health de l'ALB.
  5. Rollback Automatique : Annulation immédiate en cas d'échec de la sonde de santé.

3. Défis Relevés (Well-Architected Framework)

Défi 1 — Déploiement sans interruption (Zero-Downtime)

Mise en place d'un Instance Refresh sur l'Auto Scaling Group avec min_healthy_percentage = 100 et max_healthy_percentage = 200. Les nouvelles instances sont démarrées en parallèle et intégrées à l'ALB avant la suppression des anciennes.

Défi 2 — Rollback Automatique sur échec

Association d'une alarme CloudWatch HTTPCode_Target_5XX_Count sur l'ALB. Si le Smoke Test ou les métriques détectent une erreur 5xx, la commande cancel-instance-refresh est automatiquement déclenchée.

Défi 4 — Portes de Sécurité (Security Gates)

Intégration du plugin Maven Checkstyle / NoHTTP pour interdire l'utilisation d'URLs non sécurisées et application du principe du moindre privilège sur les rôles IAM EC2.

Défi 5 — Promotion Multi-Environnement avec Approbation

Découpage du pipeline en chaînage d'environnements (dev → staging → prod). La transition vers la production est soumise à une approbation manuelle obligatoire (Required Reviewers).

5. Difficultés Rencontrées & Solutions (Partie D)

  • Exclusion Checkstyle sur Terraform : Le plugin Checkstyle NoHTTP analysait par erreur le fichier terraform.tfstate. Solution : Ajout de **/terraform/**/* aux exclusions du pom.xml.
  • Nettoyage des ressources orphelines AWS : Des résidus d'exécutions Terraform passées provoquaient des erreurs de conflit (SGs, NAT Gateways). Solution : Écriture d'un script bash automatisé de purge complète.
  • Fédération d'accès & Secrets : Configuration sécurisée des variables d'accès AWS dans les GitHub Repository Secrets.
